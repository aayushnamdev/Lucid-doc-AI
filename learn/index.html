The 4-Step Pipeline

Every time you give Lucid a repo, it runs these 4 steps in order. This happens inside lucid/pipeline.py.

1
Clone the repo

Runs git clone --depth 1 into a temp folder. The --depth 1 means it only grabs the latest snapshot — no history. Faster, smaller. Finds every .py file, skips .venv, __pycache__, etc.

pipeline.py → subprocess.run("git clone...")
2
Build the import map

Reads every file with Python's built-in AST (Abstract Syntax Tree) — no AI here. Figures out which files import which. Labels each file as an entry point (where execution starts), core module (imported by 75%+ of other files), or internal (everything else).

pipeline.py → analyze.py → analyze_repo()
3
Write a doc for each file

8 files at a time (parallel threads). For each file, it grabs the import map data, builds a prompt, calls the AI (gpt-4.1-nano), and saves a .md file. The prompt follows Diataxis Reference format — a real structure: Public API table, Behaviour Notes, See Also links.

pipeline.py → prompts.py → llm.py → generate()
4
Write the master overview

Takes all the docs generated in Step 3, feeds them into one big prompt, and asks the AI to write an _OVERVIEW.md in Diataxis Tutorial format — narrative, goal-oriented, not a flat list. Also injects a Mermaid dependency graph (the flowchart you see in the Overview). This step uses no extra AI calls for the graph — it's built deterministically from the import map.

pipeline.py → _trim_for_synthesis() → generate() → _inject_architecture_section()

Every File — What It Does

There are 9 meaningful files. Here's what each one actually is.

cli.py WORKING Terminal entry point Run this to use Lucid from the terminal. Takes --url, --audience, --output. Calls pipeline.run() and prints progress. No browser needed.
server.py PARTIAL Web UI backend (FastAPI) Serves the website at localhost:8000. The server-side code is working. The Mermaid diagram rendering in the browser hasn't been confirmed yet — it's wired up but needs a hard-refresh test.
lucid/config.py WORKING Reads your .env file Loads LUCID_PROVIDER, LUCID_MODEL, LUCID_AUDIENCE, and the API key. If anything is missing, it stops immediately with a clear error message. Currently: OpenAI / gpt-4.1-nano.
lucid/llm.py WORKING The AI caller One job: takes a system prompt and a user prompt, calls the AI, returns the text. Has two branches: one for Anthropic, one for OpenAI/xAI (same SDK). This is the file you touch if you want to add a new AI provider (Gemini, DeepSeek, etc.).
lucid/analyze.py WORKING Import graph builder — zero AI Uses Python's AST module to read imports from every file. Builds a map of who imports who. Classifies files as entry/core/internal. Returns see_also() link data and mermaid_graph_text() for the diagram. No AI calls here — pure code analysis.
lucid/prompts.py QUALITY UNKNOWN What you tell the AI 4 builders: Reference system prompt, Reference user prompt, Tutorial system prompt, Tutorial user prompt. The prompts forbid the old "This module provides..." openers. They enforce Diataxis structure. But whether gpt-4.1-nano actually follows them well hasn't been validated beyond the first run.
lucid/pipeline.py WORKING The conductor — runs everything Calls all the other files in sequence. Manages the token budget for synthesis (_trim_for_synthesis), runs per-file generation in parallel, injects the Mermaid block into _OVERVIEW.md. Most complex file in the project.
web/index.html PARTIAL The entire web UI One HTML file, no framework, no build step. Shows the generator form, live progress checklist, and doc viewer with markdown rendering. Mermaid diagram rendering is wired but browser validation is still pending.
.env CONFIG Your API keys and settings Never committed to git. Currently active: LUCID_PROVIDER=openai, LUCID_MODEL=gpt-4.1-nano. xAI key doesn't work for chat (image-only). Anthropic has no credits.

The Quality Problem — Why You're Unhappy

You don't actually know yet

Phase 2 was built to fix the translation symptom. The first run completed on python-dotenv (21 docs). But the click comparison test — Phase 1 vs Phase 2 on the same repo — has never been run. Without that test, you're flying blind on whether the new prompts actually fixed the problem or just moved it.

These are the actual quality levers, in order of most likely to fix the feel:

1 — The prompts might not be strict enough

The Reference prompt forbids "This module provides..." and mandates a specific structure. But gpt-4.1-nano doesn't always follow strict instructions reliably. If output still feels generic, the fix is in prompts.py — more constraints, a better-structured template, or a bigger model.

2 — The model ceiling

gpt-4.1-nano is the cheapest OpenAI model. It follows formatting instructions, but not as precisely as gpt-4.1-mini or gpt-4.1. If you upgrade to gpt-4.1-mini in .env, the output quality will likely improve without changing a single line of code. Cost goes up maybe 3-5x per run.

3 — See Also links might not render

The prompt tells the AI to render See Also as markdown links with exact paths. But the LLM might hallucinate paths or skip the section. This is only verifiable by opening generated docs and checking that the links actually point to real files.

4 — The Mermaid diagram in the browser

The diagram renders in raw markdown (confirmed). Whether the web UI renders it as an SVG diagram — or shows it as a raw code block — hasn't been confirmed. This is a frontend fix, not a quality problem with the docs themselves.

5 — What's not built yet (Phase 3+)

How-to docs, Explanation docs, code examples from tests, navigation index — none of these exist. The current output is Reference + Tutorial only. That's what's in scope. Phase 3 is code validation (checking that generated param names match the real AST).

The Bigger Goal

Lucid is Phase 1 of a 10-year arc. The target is Stripe-level documentation — the gold standard — automatically generated for any repo. Stripe docs are built on Diataxis. Most doc generators produce only Reference. Lucid's gap is the full four-type suite: Reference, Tutorial, How-to, Explanation.

The honest constraint: Stripe docs are great because Stripe has 10,000 developers hitting the same walls and feeding that signal back. No AI can replicate that. Lucid can get to Stripe-level structure. The depth that comes from real user pain patterns — that requires human review on top.
